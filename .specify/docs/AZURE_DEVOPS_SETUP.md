# Azure DevOps Integration Setup Guide

This guide explains how to configure spec-kit to integrate with Azure DevOps, replacing sequential task IDs (T001, T002) with your actual Product Backlog Item (PBI) IDs from Azure DevOps.

## Overview

When enabled, the Azure DevOps integration:

1. **Fetches PBIs** from your Azure DevOps board with filtering by state and sprint
2. **Matches tasks** generated by spec-kit to your PBIs using fuzzy title matching
3. **Replaces task IDs** - instead of `T001 - Description`, you get `AB#1234 - Description`
4. **Removes user story references** - spec-kit's `[US1]` markers are replaced with PBI IDs
5. **Generates a log** - `ado-task-mapping.log` shows which tasks mapped to which PBIs

### Example Transformation

**Before (default spec-kit):**
```markdown
- [ ] [T001] Create authentication system
- [ ] [T002] [P] Initialize database schema
- [ ] [T003] [US1] Create user model in src/models/user.py
```

**After (with ADO integration):**
```markdown
- [ ] [AB#1234] Create authentication system
- [ ] [AB#1235] [P] Initialize database schema
- [ ] [AB#1236] Create user model in src/models/user.py
```

## Quick Start

### Step 1: Get an Azure DevOps Personal Access Token (PAT)

1. Go to https://dev.azure.com/YOUR-ORG (replace YOUR-ORG with your organization)
2. Click your profile icon (top right) → **Personal access tokens**
3. Click **+ New Token**
4. Configure:
   - **Name**: `spec-kit-integration`
   - **Organization**: Select your organization
   - **Expiration**: 90 days (or your preference)
   - **Scopes**: Check only **"Work Items (Read)"**
5. Click **Create**
6. **Copy the token immediately** (you won't see it again!)

### Step 2: Set Environment Variable

Set the PAT token in your PowerShell environment:

```powershell
# On Windows (PowerShell)
$env:ADO_PAT_TOKEN = 'your-pat-token-here'

# To make it permanent, add to your profile:
# In PowerShell: $PROFILE
# Add this line: $env:ADO_PAT_TOKEN = 'your-pat-token-here'

# Or set system-wide (as Administrator):
[Environment]::SetEnvironmentVariable('ADO_PAT_TOKEN', 'your-pat-token-here', 'User')
```

Verify it's set:
```powershell
$env:ADO_PAT_TOKEN  # Should show your token
```

### Step 3: Configure spec-kit

Edit `.specify/init-options.json`:

```json
{
  "ai": "copilot",
  "branch_numbering": "sequential",
  "speckit_version": "0.6.9",
  "ado": {
    "enabled": true,
    "organization": "myorg",
    "projectName": "MyProject",
    "patTokenEnvVar": "ADO_PAT_TOKEN",
    "filterByState": ["Active", "Committed"],
    "filterByIteration": "MyProject\\Sprint 25"
  }
}
```

**Configuration Reference:**

| Field | Required | Description | Example |
|-------|----------|-------------|---------|
| `enabled` | Yes | Switch to enable/disable ADO integration | `true` or `false` |
| `organization` | Yes | Azure DevOps organization name | `myorg` |
| `projectName` | Yes | Azure DevOps project name | `MyProject` |
| `patTokenEnvVar` | No | Environment variable name for PAT token | `ADO_PAT_TOKEN` |
| `filterByState` | No | Array of PBI states to include | `["Active", "Committed", "New"]` |
| `filterByIteration` | No | Iteration path (glob pattern) to filter by | `MyProject\\Sprint 25` |

### Step 4: Test the Configuration

Run the interactive setup script:

```powershell
.specify/scripts/setup-ado.ps1
```

This will:
- ✓ Validate your PAT token
- ✓ Connect to your Azure DevOps organization
- ✓ List available projects
- ✓ Show available sprints/iterations
- ✓ Test fetching PBIs with your filter criteria

### Step 5: Generate Tasks with ADO Integration

Run your normal spec-kit workflow:

```powershell
# In your feature directory
/specify
/clarify
/plan
/tasks
```

After `/tasks` completes, the `after_tasks` hook will automatically:
1. Fetch your PBIs from Azure DevOps
2. Match task descriptions to PBI titles
3. Replace T### IDs with PBI IDs
4. Generate `ado-task-mapping.log`

**Output example:**

### Step 6: Select PBI Context Before `/speckit.specify`

Use the selector script to pick one backlog item and persist context that `/speckit.specify` can consume.

```powershell
# Fast path: direct by PBI ID
.specify/scripts/powershell/select-pbi-for-specify.ps1 -PbiId 1234

# Interactive search + filters
.specify/scripts/powershell/select-pbi-for-specify.ps1 `
  -State Active,Committed `
  -Sprint "MyProject\\Sprint 25" `
  -AssignedTo "Arthur" `
  -SearchText "authentication"
```

This script supports:
- Direct lookup by PBI ID
- Filter by sprint/iteration
- Filter by assigned user
- Filter by state
- Title text search

After selection, the script writes context to `.specify/context/selected-pbi.json`.
Then run `/speckit.specify` as usual and the selected PBI metadata is included in the generated specification.

### Step 7: Create a New Backlog Item from the Spec

If you already have a specification draft and want to create a new item in Azure DevOps:

```powershell
# Preview only
.specify/scripts/powershell/create-pbi-for-specify.ps1 -DryRun -Json

# Create a new item
.specify/scripts/powershell/create-pbi-for-specify.ps1 `
  -Title "Authentication hardening" `
  -Description "Create authentication backlog item from spec context" `
  -AcceptanceCriteria "Definition of done captured in spec" `
  -Tags "security;spec-kit" `
  -Priority 2 `
  -State "To Do" `
  -Json
```

This writes context to `.specify/context/created-pbi.json` and returns the created ADO URL.

### Step 8: Push Refined Fields Back to Azure DevOps

When the linked work item section in `spec.md` is refined, sync it back to the selected ADO item:

```powershell
# Preview what will be pushed
.specify/scripts/powershell/push-pbi-refinements.ps1 -DryRun -Json

# Push the changes
.specify/scripts/powershell/push-pbi-refinements.ps1 -Json
```

The script updates non-placeholder values for:

- Description
- Acceptance Criteria
- Tags
- Story Points / Estimate

It also posts a comment to the work item documenting the refinement event.

## Troubleshooting

### Error: "PAT token not found in environment variable 'ADO_PAT_TOKEN'"

**Solution:**
1. Verify the token is set: `$env:ADO_PAT_TOKEN`
2. If empty, set it: `$env:ADO_PAT_TOKEN = 'your-token'`
3. Check your token hasn't expired in Azure DevOps

### Error: "Failed to fetch PBIs from Azure DevOps"

**Common causes:**
- Organization name is wrong → Check `dev.azure.com/YOUR-ORG`
- Project name is wrong → Verify exact spelling
- PAT token scope is insufficient → Regenerate with "Work Items (Read)" scope
- Network connectivity → Check you can reach dev.azure.com

**Debug:**
```powershell
# Check configuration loaded
$config = Get-Content .specify/init-options.json | ConvertFrom-Json
$config.ado
```

### Unmatched Tasks in Log

**Meaning:** Task description doesn't closely match any PBI title (< 70% similarity).

**Solutions:**
1. Manually verify task descriptions match PBI titles in Azure DevOps
2. Edit the task description in tasks.md to use PBI title wording
3. Re-run the hook: `.\.specify\hooks\after_tasks.ps1 -FeaturePath "..." -SpecKitRoot "."`
4. Lower similarity threshold in the module (edit azure-devops-integration.ps1, function `Find-MatchingPBI`)

### Hook Doesn't Run Automatically

**Check 1:** Is ADO enabled?
```json
"ado": {
  "enabled": true,  // Must be true
  ...
}
```

**Check 2:** Does the hook file exist?
```powershell
Test-Path .\.specify\hooks\after_tasks.ps1
```

**Check 3:** Run manually for debugging:
```powershell
.\.specify\hooks\after_tasks.ps1 -FeaturePath "specs/001-my-feature" -SpecKitRoot "."
```

## How It Works

### Architecture

1. **Module** (`.specify/modules/azure-devops-integration.ps1`):
   - Handles Azure DevOps REST API authentication
   - Fetches PBIs with state and iteration filtering
   - Performs fuzzy string matching to map tasks to PBIs

2. **Hook** (`.specify/hooks/after_tasks.ps1`):
   - Runs automatically after `/tasks` agent completes
   - Uses the module to fetch and match PBIs
   - Rewrites tasks.md with PBI IDs
   - Generates mapping log

3. **Configuration** (`.specify/init-options.json`):
   - Stores organization, project, and filter settings
   - `enabled` flag allows quick on/off
   - Graceful fallback if ADO config missing

### Fuzzy Matching Algorithm

Tasks are matched to PBIs using **Levenshtein distance** (edit distance):

- Compares task description to PBI title
- Calculates similarity percentage
- Default threshold: **70% minimum similarity** for a match
- **First pass matching**: Tries to match by title
- **Unmatched tasks**: Kept with original T### ID and logged

**Example:**
- PBI Title: "Create user authentication system"
- Task: "T001 Create user authentication"
- Similarity: ~92% ✓ MATCH

- PBI Title: "Fix login button styling"
- Task: "T005 Implement OAuth provider"
- Similarity: ~35% ✗ NO MATCH

## Advanced Configuration

### Custom Similarity Threshold

Edit `.specify/modules/azure-devops-integration.ps1`, function `Find-MatchingPBI`:

```powershell
$bestScore -ge $SimilarityThreshold = 80  # Raise to 80% for stricter matching
```

### Filter by Custom Fields

The module currently filters by State and Iteration. To add custom filters (e.g., by Area, tags):

Edit `.specify/modules/azure-devops-integration.ps1`, function `Get-AzureDevOpsPBIs`:

```powershell
if ($FilterByArea) {
    $query += " AND [System.AreaPath] UNDER '$FilterByArea'"
}
```

Then add to `init-options.json`:
```json
"filterByArea": "MyProject\\BackendTeam"
```

## FAQs

**Q: Does this sync changes back to Azure DevOps?**
A: No, it's one-way only (ADO → spec-kit tasks). Changes to tasks.md don't update Azure DevOps. That's a future enhancement.

**Q: Can I use a service principal instead of PAT?**
A: Not currently - the module uses basic auth with PAT. Service principal support is on the roadmap.

**Q: What if a PBI title changes after tasks are generated?**
A: Re-run the hook manually with the same feature path to remap with updated PBI data.

**Q: Can I disable Azure DevOps integration temporarily?**
A: Yes, set `"enabled": false` in init-options.json and tasks.md won't be modified on next `/tasks` run.
